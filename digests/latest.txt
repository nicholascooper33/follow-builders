Latent Space — "Notion's Token Town: 5 Rebuilds, 100+ Tools, MCP vs CLIs and the Software Factory Future" (with Simon Last & Sarah Sachs of Notion)

Takeaway: Building production AI agents at scale is mostly about knowing when you're swimming upstream against model limitations versus when the river is ready — and Notion's edge is having rebuilt their agent infrastructure five times and surviving to tell the story.

Simon Last (engineer) and Sarah Sachs (product) from Notion walk through how the company has been building AI agents since late 2022, starting with GPT-4 before function calling even existed. They designed their own tool-calling framework, tried to fine-tune models to use it, and it consistently failed to be robust enough — until roughly Sonnet 3.6 in early 2025. That unlock is what led to Notion's agent and then custom agents shipping publicly.

A key insight from Sarah: the two skills that matter most at the frontier are (1) quickly recognizing when you're "swimming upstream" against model limitations rather than fighting product or infrastructure problems, and (2) knowing which direction the river is flowing so you can build ahead. They've had multiple failed versions of capabilities — transcription before meeting notes, multiple custom agent rebuilds — and the discipline is knowing when to stop pressing and wait for the model to catch up.

Simon flags that in his view, "coding agents are the kernel of AGI — everything is a coding agent." The next big bet he's excited about is what he calls the "software factory": a maximally automated workflow for developing, debugging, merging, reviewing, and maintaining a codebase with multiple agents working together. Notion is actively building toward this.

On MCP vs CLIs, Simon says he's bullish on both: MCPs for narrow, lightweight agents where you want a strong permission model and don't need a full coding agent runtime; CLIs for broader use. Sarah adds that Notion is committed to supporting their MCP regardless of internal perspective, because their users expect it.

Direct quote from Simon Last: "I think coding agents are the kernel of AGI. Everything is a coding agent... Your agent can bootstrap its own software and capabilities and actually debug and maintain them."

Anthropic Engineering — "Scaling Managed Agents: Decoupling the brain from the hands"
A deep technical post on how Anthropic built Managed Agents by virtualizing agent components — session (the append-only log), harness (the loop that calls Claude), and sandbox (execution environment) — so each can be swapped independently as models improve. The key insight: agent harnesses encode assumptions about model limitations that go stale fast (e.g., "context anxiety" workarounds for Sonnet 4.5 became dead weight with Opus 4.5). By decoupling components like an OS virtualizes hardware, the interfaces stay stable while implementations underneath evolve freely. The post is a direct argument against the "pet server" anti-pattern — a coupled single-container approach where losing the container means losing the session.
https://www.anthropic.com/engineering/managed-agents

Claude Blog — "Claude Managed Agents: get to production 10x faster"
Anthropic's public launch of Claude Managed Agents (public beta on the Claude Platform): a suite of composable APIs for building and deploying cloud-hosted agents without having to build sandboxing, state management, permissioning, checkpointing, or context recovery from scratch. The harness handles tool calls, context management, and error recovery. It supports long-running sessions (hours, even through disconnections), multi-agent coordination where agents spin up and direct other agents (research preview), and trusted governance with scoped permissions and execution tracing built into Claude Console. In internal testing on structured file generation, Managed Agents improved task success by up to 10 points over a standard prompting loop, with the biggest gains on the hardest problems.
https://claude.com/blog/claude-managed-agents
